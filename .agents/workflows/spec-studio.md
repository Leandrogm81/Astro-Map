# SPEC STUDIO - AstroMap

**Descrição:**  
Use este workflow quando a necessidade principal for pensar, organizar e documentar uma ideia antes da implementação.  
O objetivo é transformar uma ideia vaga, parcial ou já discutida em um artefato claro de planejamento, sem iniciar execução.

---

## 🎯 Missão

Converter intenção em clareza, escopo e documentação utilizável.

Este workflow existe para:

- entender o problema real
- definir objetivo e não-objetivos
- separar MVP de expansão futura
- reduzir ambiguidade
- gerar um documento de planejamento sólido
- preparar uma base para etapas futuras, sem executar o projeto

---

## 🧭 Princípios

- **Planejamento vem antes da execução.**
- **Documento bom reduz retrabalho.**
- **Escopo claro vale mais do que documento longo.**
- **Não-objetivos são obrigatórios.**
- **MVP deve ser separado da visão futura.**
- **Especificação não é implementação.**
- **Se ainda houver ambiguidade crítica, o workflow deve primeiro clarificar, não fingir certeza.**

---

## 📥 Quando Acionar

Acione este workflow quando você quiser:

- transformar uma ideia em PRD
- montar uma spec de produto, projeto ou funcionalidade
- definir MVP
- organizar escopo
- esclarecer prioridades
- mapear usuários e necessidades
- gerar um documento-base antes de BUILD FLOW
- pensar o projeto antes da execução

---

## 📥 Entradas Esperadas

Sempre que possível, informar:

- ideia inicial
- problema que deseja resolver
- para quem isso existe
- objetivo principal
- contexto do projeto
- referências, prints, links ou exemplos
- limitações conhecidas
- urgência percebida
- tipo de documento desejado, se já souber

Mesmo com entradas incompletas, o workflow deve ajudar a estruturar o raciocínio.

---

## 📤 Saídas Possíveis

Este workflow pode gerar:

- **Discovery Brief**
- **PRD**
- **Product Spec**
- **Feature Spec**
- **Project Spec**
- **MVP Scope**
- **Execution Brief**
- **Open Questions List**

A saída deve ser escolhida de acordo com o nível de maturidade da ideia e com a necessidade do projeto.

---

## 🔀 Modos Internos

### DISCOVERY MODE

Use quando a ideia ainda estiver crua, confusa ou ampla demais.

Este modo serve para:

- esclarecer o problema real
- separar desejo de necessidade
- entender usuário e contexto
- levantar dúvidas críticas
- decidir se já existe clareza suficiente para seguir

Este modo responde melhor à pergunta:
> “O que realmente vale a pena definir aqui antes de escrever uma spec?”

---

### PRD MODE

Use quando o foco principal for:

- problema
- usuário
- valor do produto
- escopo
- MVP
- critérios de sucesso
- priorização

Este modo responde melhor à pergunta:
> “O que estamos criando, para quem, por quê e com qual resultado esperado?”

---

### MVP MODE

Use quando a ideia já faz sentido, mas o risco principal for **escopo excessivo**.

Este modo serve para:

- cortar excesso
- separar essencial de desejável
- definir primeira versão
- registrar o que fica para depois
- proteger contra overbuilding

Este modo responde melhor à pergunta:
> “Qual é a menor versão valiosa que vale construir agora?”

---

### SPEC MODE

Use quando o foco principal for:

- definição operacional da solução
- comportamento esperado
- requisitos
- fluxos
- restrições
- critérios de validação
- dependências

Este modo responde melhor à pergunta:
> “Como essa funcionalidade ou projeto deve ser definido para poder ser construído depois?”

---

## 🧠 Regra de Escolha de Modo

Escolha o modo assim:

- Se a ideia ainda estiver ampla, vaga ou com muitas perguntas → **DISCOVERY MODE**
- Se a ideia estiver mais em “produto/visão/necessidade” → **PRD MODE**
- Se a principal dificuldade for delimitar a primeira versão → **MVP MODE**
- Se a ideia já estiver mais em “funcionalidade/fluxo/requisito” → **SPEC MODE**
- Se ainda faltar clareza crítica → gerar primeiro uma **Open Questions List**
- Se o usuário pedir explicitamente PRD → **PRD MODE**
- Se o usuário pedir explicitamente MVP → **MVP MODE**
- Se o usuário pedir explicitamente Spec → **SPEC MODE**

---

## 🛠️ Protocolo de Execução

### Passo 1: Clarificar a Ideia

Transformar a ideia em uma frase simples:

> “Queremos criar / melhorar / lançar [X] para resolver [Y] para [Z].”

Se isso ainda não estiver claro, o workflow deve fazer perguntas curtas e úteis antes de gerar o documento final.

---

### Passo 2: Identificar o Problema Real

Responder:

- qual problema isso resolve?
- para quem esse problema existe?
- o problema é recorrente, relevante ou apenas desejável?
- o que acontece hoje sem essa solução?

---

### Passo 3: Definir Usuário e Contexto

Mapear:

- usuário principal
- usuário secundário, se existir
- contexto de uso
- momento de uso
- frequência provável
- ambiente ou canal onde isso será usado

---

### Passo 4: Declarar Objetivos e Não-Objetivos

Definir com clareza:

- objetivo principal
- objetivos secundários
- não-objetivos
- o que está fora do escopo

Sem isso, o documento tende a ficar inflado e ambíguo.

---

### Passo 5: Separar Essencial de Futuro

Responder:

- o que é indispensável?
- o que é desejável, mas não necessário agora?
- o que pode esperar?
- o que é visão futura?

---

### Passo 6: Mapear Requisitos

Organizar os requisitos por grupos, conforme o caso:

- funcionais
- experiência do usuário
- regras de negócio
- dados
- integrações
- restrições técnicas
- critérios de validação
- critérios de sucesso

---

### Passo 7: Registrar Riscos, Dependências e Dúvidas

Identificar:

- riscos principais
- dependências
- ambiguidades
- decisões pendentes
- perguntas em aberto

Se a incerteza ainda for alta, o workflow pode gerar primeiro uma **Open Questions List** antes do documento final.

---

### Passo 8: Selecionar o Modo Final

Escolher entre:

- **DISCOVERY MODE**
- **PRD MODE**
- **MVP MODE**
- **SPEC MODE**
- **Open Questions List**, se ainda faltar clareza

---

### Passo 9: Gerar o Documento

Depois da clareza mínima, gerar o artefato final no formato adequado.

O documento deve ser:

- claro
- organizado
- enxuto
- útil
- sem implementação prematura
- pronto para servir de base para workflows posteriores

## 🧱 Estrutura de Saída — DISCOVERY MODE

Quando o modo escolhido for **DISCOVERY MODE**, usar preferencialmente esta estrutura:

1. Título
2. Ideia resumida
3. Problema percebido
4. Usuário potencial
5. Hipótese de valor
6. O que já está claro
7. O que ainda está ambíguo
8. Principais perguntas
9. Riscos de entendimento
10. Próximo passo sugerido
11. Modo recomendado depois da descoberta

---

## 🧱 Estrutura de Saída — PRD MODE

Quando o modo escolhido for **PRD MODE**, usar preferencialmente esta estrutura:

1. Título
2. Resumo Executivo
3. Problema
4. Objetivo
5. Usuário-alvo
6. Contexto de uso
7. Escopo
8. Não-objetivos
9. MVP
10. Requisitos principais
11. Critérios de sucesso
12. Riscos
13. Dependências
14. Questões em aberto

---

## 🧱 Estrutura de Saída — MVP MODE

Quando o modo escolhido for **MVP MODE**, usar preferencialmente esta estrutura:

1. Título
2. Objetivo da primeira versão
3. Problema que o MVP resolve
4. Usuário principal
5. Escopo mínimo viável
6. O que entra no MVP
7. O que não entra no MVP
8. Critérios de prontidão
9. Riscos de escopo
10. Dependências
11. Próximas expansões possíveis
12. Questões em aberto

---

## 🧱 Estrutura de Saída — SPEC MODE

Quando o modo escolhido for **SPEC MODE**, usar preferencialmente esta estrutura:

1. Título
2. Objetivo da funcionalidade/projeto
3. Escopo
4. Fora de escopo
5. Fluxo esperado
6. Requisitos principais
7. Regras e restrições
8. Dados e contratos relevantes
9. Critérios de validação
10. Dependências
11. Riscos
12. Questões em aberto

---

## 🧱 Estrutura de Saída — Open Questions List

Quando ainda não houver clareza suficiente para fechar um documento:

1. Ideia resumida
2. O que já está claro
3. O que ainda está ambíguo
4. Perguntas críticas
5. Próximo passo recomendado
6. Modo sugerido depois da clarificação

---

## 🚫 Fronteiras

Este workflow:

- **não implementa**
- **não escreve código**
- **não inicia BUILD FLOW automaticamente**
- **não detalha arquitetura cedo demais**
- **não faz DEBUG TRACE**
- **não atualiza MAESTRO**
- **não cria log por reflexo**
- **não confunde planejamento com execução**
